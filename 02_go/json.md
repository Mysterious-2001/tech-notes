# 一、JSON 是什么？

**JSON（JavaScript Object Notation）** 是一种：

- 轻量级数据交换格式
- 纯文本格式
- 跨语言通用
- 结构化数据表示方式

它本质就是：**一种数据结构的字符串表示方式**

---

## JSON 长什么样？

JSON 只支持 6 种数据类型：

```
{
  "name":"Alice",
  "age":18,
  "isStudent":true,
  "scores": [90,88,95],
  "address": {
    "city":"Beijing",
    "zip":"100000"
  },
  "extra":null
}
```

支持的数据类型：

|类型|示例|
|---|---|
|string|`"hello"`|
|number|`123`、`3.14`|
|boolean|`true` / `false`|
|null|`null`|
|array|`[1,2,3]`|
|object|`{ "a":1 }`|

---

# 二、JSON 的核心设计理念

JSON 只有两种结构：

### 1️⃣ 对象（Object）

```
{
  "key":"value"
}
```

- 本质是 **key-value**
- key 必须是字符串
- 必须用双引号

---

### 2️⃣ 数组（Array）

```
[1,2,3]
```

- 有序集合
- 可以嵌套对象或数组

---

# 三、JSON 在工程中的用途

### 1️⃣ 前后端接口传输


### 3️⃣ 数据库存储（NoSQL）

# 四、JSON 的工程注意事项（重点）

这是很多人踩坑的地方。

---

## ⚠️ 1. 必须使用双引号

❌ 错误：

```
{
  name: "Alice"
}
```

✅ 正确：

```
{
  "name":"Alice"
}
```

JSON 标准规定：

**字符串和 key 必须使用双引号。**

---

## ⚠️ 2. 不能有尾逗号

❌ 错误：

```
{
  "name":"Alice",
}
```

❌ 错误：

```
[1,2,3,]
```

✅ 正确：

```
{
  "name":"Alice"
}
```

---

## ⚠️ 3. 不支持注释

JSON 不允许：

```
{
  // 用户名
  "name":"Alice"
}
```

解决方案：

- 使用 JSON5
- 或使用 yaml
- 或使用额外字段说明

---

## ⚠️ 4. 数字精度问题（非常重要）

在 JS / Go / Java 里：

```
{
  "id":9007199254740993
}
```

在 JavaScript 中会精度丢失：

```
9007199254740992
```

因为 JS 使用 double 浮点数。

解决方案：

✅ 把大整数转为字符串：

```
{
  "id":"9007199254740993"
}
```

这是金融系统常见做法。

---

## ⚠️ 5. 空值处理

很多人混淆：

- `null`
- 不存在字段
- 空字符串
- 空数组

例如：

```
{
  "name":null
}
```

和：

```
{
}
```

语义完全不同。

接口设计时必须明确：

- 是否允许字段缺失？
- 是否允许 null？
- 默认值是什么？

---

## ⚠️ 6. 安全问题

### 1️⃣ 反序列化攻击

如果语言支持自动反序列化成对象（如 Java），要小心：

- 不要反序列化未知来源 JSON 到复杂对象
- 防止 gadget chain 攻击

---

### 2️⃣ JSON 注入

拼接 JSON 时不要这样：

```
jsonStr:="{ \\"name\\": \\""+input+"\\" }"
```

必须使用标准库编码：

```
json.Marshal()
```

---

## ⚠️ 7. 字段大小写敏感

```
{
  "Name":"Alice"
}
```

和

```
{
  "name":"Alice"
}
```

很多语言区分大小写。

---

## ⚠️ 8. 时间格式统一

JSON 没有时间类型。

常见规范：

```
ISO8601
```

例如：

```
2026-03-03T11:20:00Z
```

建议：

- 后端统一用 UTC
- 明确时区
- 不要用时间戳和字符串混用

# 七、工程级最佳实践总结

我给你一个“工程铁律清单”：

### ✅ 设计层面

- 字段命名统一（snake 或 camel）
- 明确 null 语义
- 时间格式统一
- 大整数用字符串

---

### ✅ 编码层面

- 永远使用官方 JSON 库
- 不手写 JSON 字符串
- 不信任客户端数据
- 限制反序列化类型

---

### ✅ 性能层面

- 大数据避免深层嵌套
- 避免循环引用
- 大文件使用流式解析（stream decode）

---

# 八、一个高级思考（架构层）

JSON 是：

> 结构化数据的“最低公分母”

它优点是通用，但缺点是：

- 无 schema
- 无类型约束
- 无版本管理

大型系统通常会：

- 使用 JSON Schema
- 使用 Protobuf 代替 JSON
- 或者用 Avro / Thrift

当系统变大，你会发现 JSON 不够严谨。

---

# 最后总结一句话

> JSON 是简单的，但工程化使用绝对不简单。

真正的能力不在于“会写 JSON”，

而在于“设计一个长期可扩展的 JSON 协议”
