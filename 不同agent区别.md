不同 Agent 类型的核心差异体现在**对工具参数的支持能力、适配的大模型范围、是否支持原生工具调用、功能特性**四个维度，下面是最常用的 4 类 Agent 的具体区别，你可以直接根据自己的场景选型：

---

## 1. 基础 ReAct 代理（`create_react_agent`）

### 核心定位

  

最经典、兼容性最强的通用入门代理

### 核心特点

  

- 完全靠 Prompt 引导大模型遵循「思考→调用工具→看结果→再思考」的 ReAct 推理流程
- 仅支持**单参数工具**（比如工具只需要传一个搜索词 / 城市名），一次只能调用 1 个工具
- 默认不支持对话记忆，需要自己额外在 Prompt 里拼接历史对话

### 适配模型

  

所有能正常生成文本的大模型，**不需要模型有原生工具调用能力**，哪怕是能力较弱的开源小模型也能用

### 适用场景

  

简单 Demo 测试、工具都是单参数、用的是没有工具调用能力的开源小模型

### 缺点

  

不支持多参数工具，大模型能力弱时容易输出不符合格式的内容，解析报错率高

---

## 2. 结构化聊天代理（`create_structured_chat_agent`）

### 核心定位

  

支持多参数工具、带对话记忆的通用代理

### 核心特点

  

- Prompt 要求大模型输出固定格式的 JSON 结构来调用工具，原生支持**多参数工具**（比如订酒店工具需要传「城市、入住时间、人数」三个参数）
- 原生适配对话记忆，不用额外改 Prompt 就能保留上下文
- 一次只能调用 1 个工具

### 适配模型

  

所有能正常生成文本的大模型，**不需要模型有原生工具调用能力**

### 适用场景

  

工具需要传多个参数、需要保留对话上下文、用的是没有工具调用能力的模型

### 缺点

  

大模型能力弱时容易输出非法 JSON，解析报错率比原生工具调用代理高

---

## 3. OpenAI 工具代理（`create_openai_tools_agent`）

### 核心定位

  

专门适配 OpenAI 系列模型原生工具调用能力的代理

### 核心特点

  

- 不依赖 Prompt 约束格式，直接调用 OpenAI 官方的 Function Calling 接口，让模型按照工具的定义自动输出符合规范的参数，**解析错误率几乎为 0**
- 支持单 / 多参数工具，支持**一次并行调用多个工具**（比如同时查天气 + 订机票，一轮就搞定，不需要分两次调用）
- 原生支持对话记忆

### 适配模型

  

仅支持 OpenAI 的`gpt-3.5-turbo`、`gpt-4`等带原生工具调用能力的模型

### 适用场景

  

用 OpenAI 模型的所有生产场景，是 OpenAI 用户的首选

### 缺点

  

仅支持 OpenAI 模型，通用性差

---

## 4. 通用工具调用代理（`create_tool_calling_agent`）

### 核心定位

  

适配所有带原生工具调用能力大模型的通用代理

### 核心特点

  

- 是 OpenAI 工具代理的通用版，逻辑和 OpenAI 工具代理完全一致，但做了通用适配，支持所有符合 LangChain 工具调用规范的大模型（比如字节豆包、Anthropic Claude、阿里通义千问、百度文心一言等都支持）
- 同样支持单 / 多参数、并行调用多个工具、解析错误率极低、支持对话记忆

### 适配模型

  

所有带原生工具调用能力的大模型

### 适用场景

  

用非 OpenAI 的商用 / 开源大模型，且模型有原生工具调用能力的生产场景，是现在的主流首选方案

### 缺点

  

不支持没有原生工具调用能力的老模型

---

## 快速选型表

  

|你的场景|首选 Agent 类型|
|---|---|
|用 OpenAI 系模型|`create_openai_tools_agent`|
|用其他带原生工具调用能力的模型（豆包 / Claude / 通义等）|`create_tool_calling_agent`|
|用的模型没有原生工具调用能力，且工具是多参数 / 需要对话记忆|`create_structured_chat_agent`|
|用的模型没有原生工具调用能力，工具都是单参数，只做简单 Demo|`create_react_agent`|
